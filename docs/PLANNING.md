# Планирование: динамический фуд-кост, LTV/чек, списания, «что если»

Каркас под четыре функции мини-ERP поверх существующего ядра. Все модули **чистые**
(Decimal, без БД и сети) и проверяются offline, как и весь проект. БД-таблицы рецептов
сознательно НЕ заводим сейчас — это решение зафиксировано в [ARCHITECTURE.md](ARCHITECTURE.md) §6
(ждём каталог товаров из Эвотора и цены продажи от владельца).

Запуск проверки:
```bash
.venv/bin/python -m backend.costing.demo     # тех-карты + списания
.venv/bin/python -m backend.scenarios.demo   # LTV/чек + «что если» + break-even
```

## 1. Динамический фуд-кост и тех-карты — `backend/costing/`

**Задача:** при изменении закупочной цены молока/зерна себестоимость каждого напитка
пересчитывается автоматически.

**Источник:** дозы, закупочные цены и фасовка взяты из самого `Расчет фуд-коста
напитков_ кофе и чай.xlsx` (лист «Расчет фуд-коста напитков кофе»): базовые позиции
250/350/450 мл (A14:E54) и лист «Добавки» (A56:E64). Колонка F «Себестоимость пересчёт»
в Excel заведена, но **пустая** — её и закрывает динамический движок.

**Что сделано:**
- `ingredients.py` — единый справочник закупочных цен. Цена хранится в исходном виде
  (молоко 94 ₽/л) и переводится в ₽ за дозируемую единицу (0.094 ₽/мл). Поменять цену →
  пересчитаются все тех-карты. Включает допы (какао, матча, сироп, сливки, арахис…).
  `bump_price`/`with_price`/`price_book` — для сценариев «поставщик поднял цену».
- `techcard.py` — рецепт = список `Component(ингредиент, доза)`; `TechCard.cost(prices)`
  считает себестоимость **на лету** из прайса. Базовые позиции + допы (`drink_card`,
  `addon_cost`). `recompute_on_price_change` показывает, как сдвиг цены меняет позиции.
  `reconcile`/`reconcile_addons` сверяют динамику с замороженными значениями Excel
  (`cost_reference.CONTROL`/`ADDONS`) и раскладывают расхождение по компонентам.

**Проверено:** базовые позиции дают 68.92/78.55/80.34 ₽; расхождение с Excel-контролем
(+1.10 ₽) раскладывается ровно в Находку 3 (зерно −0.60 ₽ округление, корица +1.70 ₽ спор
по цене). Молоко +10% → 250 мл 68.92 → 70.61 ₽ (+1.69 = 180 мл × 0.0094).

**Новые находки по допам (сверка `reconcile_addons`):**
- какао / порошки / сироп — сходятся с Excel ✅.
- **матча −4.28 ₽ (находка, подтверждена владельцем)**: доза 3 г → верная себестоимость
  6.47 ₽, а в Excel зафиксировано 10.75 ₽ — завышение на 4.28 ₽/порцию. Динамический
  движок считает верно (6.47 ₽); значение в Excel ошибочно.
- ванильный сахар +0.06 ₽ — мелкое расхождение округления.
- **сливки 53.73 ₽** и **арахисовая паста 10.80 ₽** — в Excel себестоимость порции не
  посчитана (`ADDONS_NO_COST`), движок её добил из цены и дозы. Сливки 270 мл дорогие —
  это фактически порция РАФа.
- В «шапочной» тех-карте Excel (K4:N9) есть строка **«Вода / Потери» = 0.58 ₽/чашка** —
  потери уже зашиты в себестоимость → прямой мостик к модулю списаний (§3).

## 2. Калькулятор LTV и среднего чека — `backend/scenarios/menu.py`

**Задача:** видеть, как ввод выпечки/сиропов влияет на средний чек и маржинальность.

**Что сделано:**
- `MenuItem` (цена продажи + себестоимость из тех-карты → маржа), `Menu` (структура продаж).
- `metrics()` — выручка, средняя цена позиции, маржа; помечает позиции без цены/себестоимости.
- `impact_of_addon` — доп-апсейл (сироп): доливается к доле текущих чеков, число чеков не
  меняется → средний чек и маржа растут. `impact_of_new_item` — новая позиция (выпечка),
  с опцией «приносит собственные новые чеки».
- `ltv()` — маржинальная LTV: чек × визитов/мес × горизонт × маржа (с явными допущениями).

**Проверено:** сироп к 30% позиций (+50 ₽) → средний чек 361.43 → 386.07 ₽ (+24.64),
маржа +1.4 п.п., прибыль +15 180 ₽.

**Что нужно для прода:**
- [ ] **Цены продажи по позициям** от владельца (Эвотор их не отдаёт) — главный блокер
      (тот же, что для маржи по товарам в основном роадмапе).
- [ ] Частота визитов и горизонт удержания для LTV — из истории чеков Эвотора (когортно).

## 3. Управление потерями (списание) — `backend/costing/writeoffs.py`

**Задача:** оценивать пролитый при настройке помола эспрессо и скисшее молоко в деньгах.

**Что сделано:**
- `WriteOff` (период, ингредиент, объём, причина) → стоимость из того же прайса, что тех-карты.
- `WriteOffReason` — калибровка/скисшее молоко/просрочка/бой/обучение/прочее.
- `estimate_calibration` (N проливов × доза × дни), `estimate_soured_milk` (литры) — прикидки,
  пока нет точного учёта.
- `summarize` — итог по причинам/ингредиентам + доля в выручке и COGS.

**Проверено:** 4 пролива/день × 17 г × 30 дн + 8 л молока + бой → 5 386 ₽/мес
(1.8% выручки, 21.5% COGS) — деньги, которых нет в текущем P&L.

## 4. Моделирование «что если» + точка безубыточности — `backend/scenarios/whatif.py`

**Задача:** ввести «аренда +15%, трафик −10%» и увидеть новый P&L и когда наступает break-even.

**Что сделано:**
- `Scenario` (множители: трафик, цена, COGS-удорожание; множители/добавки по статьям) +
  `Scenario.from_pct` (из процентов). `transform` применяет сдвиги, понимая, что переменная
  себестоимость растёт с трафиком, а не с ценой продажи.
- `apply` считает базовый и сценарный P&L **через существующий `ProfitCalculator`** + сравнение.
- `break_even` — маржинальным методом: безубыточная выручка и запас по трафику (на сколько %
  может просесть поток до нуля прибыли).

**Проверено:** аренда +15%, трафик −10% поверх мая → прибыль 101 455 → 60 626 ₽,
маржа 33.1 → 22.0%, остаётся прибыльным; безубыточная выручка 196 032 → 209 837 ₽.

**Что нужно для прода:**
- [ ] Встроить в бота кнопкой «Что если» (готовые пресеты сдвигов + ввод своих).
- [ ] Базу сценария брать из honest-P&L месяца, а не из иллюстративного словаря.

## Где это живёт в архитектуре

```
backend/
├── costing/                 # №1 тех-карты, №3 списания
│   ├── ingredients.py       #   справочник закупочных цен (единый источник)
│   ├── techcard.py          #   динамическая себестоимость + сверка с Excel
│   ├── writeoffs.py         #   списания → деньги → доля в COGS
│   └── demo.py
└── scenarios/               # №2 LTV/чек, №4 «что если»
    ├── menu.py              #   средний чек, маржа, эффект допа/позиции, LTV
    ├── whatif.py            #   Scenario + ProfitCalculator + break-even
    └── demo.py
```

Принципы соблюдены: деньги — `Decimal`; расчёт прибыли — только через `ProfitCalculator`;
себестоимость — поверх существующего `cost_reference` (не дублируя, а сверяясь с ним);
модули чистые и проверяются offline. Следующий шаг материализации в БД и UI — общий с
основным роадмапом блокер «каталог Эвотора + цены продажи».
